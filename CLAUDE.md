# Archive Repository 가이드

이 저장소는 학습 기록(TIL)을 한국어로 정리하는 개인 노트다. Claude는 다음 5가지 작업에 활용한다.

| 작업 | Slash command | 담당 agent |
|---|---|---|
| 사전 리서치 | `/research <주제>` | til-researcher |
| 글 초안 작성 | `/write <주제>` | til-writer |
| 사실관계 검증 | `/factcheck <파일>` | til-factchecker |
| 코드 스니펫 작성 | `/snippet <요청>` | til-snippet-writer |
| 글 리뷰·첨삭 | `/review <파일>` | til-reviewer |

---

## 파일명 규칙

- 한글 그대로 사용한다.
- 공백은 하이픈(`-`)으로 치환한다.
- `?`, `:`, `/`, `*` 등 OS·URL 특수문자는 사용하지 않는다.
- 확장자는 `.md`.

예시:
```
리액트-서버-컴포넌트.md
B-트리와-B+트리.md
postgres-deadlock-원인-분석.md
```

## 글 frontmatter 템플릿

모든 TIL 파일은 다음 frontmatter로 시작한다.

```yaml
---
title: 글 제목
date: 2026-05-01
tags: [react, ssr]
sources:
  - https://react.dev/reference/...
---
```

- `title`: 글 본문 H1을 대체한다(중복 H1 작성 금지).
- `date`: ISO 8601(YYYY-MM-DD). 최초 작성일.
- `tags`: 소문자 영문 키워드 배열. 카테고리 폴더와 별개로 횡단 검색용.
- `sources`: 본문에서 참조한 1차 출처 URL 목록.

## 글쓰기 톤·스타일

- **언어**: 한국어. 평어체(`~다`) 기본. 1인칭 학습기록 톤(예: "이번에 처음 알았다", "이 repo의 케이스에선 ~다").
- **종결어미**: `~다`로 통일. `~요`·`~습니다` 혼용 금지.
- **헤딩**: H1은 frontmatter title이 대체하므로 본문은 H2(`##`)부터 시작.
- **표준 섹션 순서** (필요한 것만 사용):
  1. 배경 — 왜 이 주제를 보게 됐는지
  2. 핵심 개념 — 정의·원리
  3. 예시 / 코드 — `/snippet` 활용
  4. 주의점·한계 — 흔한 오해, 실패 케이스
  5. References — 인용한 출처
- **코드 블록**: 언어 태그 필수(```ts, ```py, ```sh 등). 언어/런타임 버전이 결과에 영향을 주면 본문에 명시.

---

## 출처 인용 규칙

이 섹션이 출처 정책의 정본이다. 모든 agent·skill은 여기를 따른다.

- **1순위**: 공식문서(MDN, 공식 docs), RFC, 논문, 표준 스펙, 언어/프레임워크 spec.
- **2순위**: 작성자가 명확한 기술 블로그, 컨퍼런스 발표.
- **보조**: Stack Overflow, GitHub issue/discussion, 커뮤니티 답변 (다른 출처와 cross-check된 경우만).

모든 사실 주장(fact claim)에는 출처를 표기한다. 두 가지 방식 중 택일:

1. 인라인 링크: 문장 끝에 [(MDN)](https://developer.mozilla.org/...) 형태.
2. 본문 끝 References 섹션에 번호 매김 후 본문에 `[1]` 형태로 참조.

주관적 의견·경험담은 출처 없이 작성 가능하되, 그 사실을 본문에서 명시("내 경험상", "이 repo의 케이스") 한다.

## Claude 사용 가이드

### ⚠️ 필수 규칙: 항상 에이전트 프로세스를 거쳐야 한다

**모든 TIL 글은 어떤 세션, 어떤 질문이든 절대 직접 작성하면 안 되고, 반드시 아래 프로세스를 따라야 한다:**

1. **`/research <주제>`** → til-researcher 에이전트로 리서치 수행
2. **`/write <주제>`** → til-writer 에이전트로 초안 작성
3. **필요시 다듬기** → `/factcheck`, `/review`, `/snippet` 등으로 품질 향상

이 프로세스를 거치지 않으면 검증 없이 부정확한 내용이 작성될 수 있고, 프로젝트 품질 기준을 무시하는 것이다.

---

### `/research <주제>`
글 쓰기 **전** 단계. 공식문서 우선으로 자료조사하고 핵심 개념·인용 가능한 fact·추천 글 구조를 정리한 노트를 받는다.

### `/write <주제>`
**"1 What + 5 Whys"** 6-섹션 구조(정의 → 탄생배경 → 동작원리 → 장단점 → 비교 → 사용시점)로 완성된 TIL 초안을 작성한다. 리서치 노트가 있으면 참고할 수 있다. 이후 `/review` / `/snippet` / `/factcheck`로 다듬는다.

### `/snippet <요청>`
본문에 들어갈 코드 예제를 만든다. self-contained, 동작 검증 우선. 주석은 "why"만.

### `/factcheck <파일>`
이미 작성한 글의 사실관계를 검증한다. 표 형식(주장/판정/근거 URL/수정안)으로 결과를 받는다.

### `/review <파일>`
구조·명료성·문법 위주의 첨삭. 사실관계는 다루지 않으므로 의심되면 `/factcheck`를 따로 돌린다.

### Slash command 없이 호출
"이 주제 리서치해줘", "이 글 팩트체크 부탁", "이 부분 코드 예제 만들어줘", "이 글 리뷰해줘" 같은 자연어로도 같은 skill이 발동한다.
